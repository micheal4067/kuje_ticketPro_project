Settings

The Settings section in TicketPro allows administrators to configure system-wide options such as password management, communication settings, and maintenance tasks.

Communication Settings

Communication settings control how TicketPro interacts with entry and exit devices as well as parking slot management. Each entity has its own database and its own set of communication settings.

How it Works

  • The system fetches all available entity’s from the central ticket database.
  • You select an (entity) from the dropdown list. This determines which entity database to use.
  • The system checks for the communication settings table in that database.
  • If it exists, it loads the current configuration values (IPs and parking slots).
  • Admins can edit and save:
    • Entry 1 & 2 IPs: IP addresses of entry barriers.
    • Exit 1 & 2 IPs: IP addresses of exit barriers.
    • Total Parking Slots: Maximum slot capacity for the entity.
  • When saved, the system updates the record in the selected entity database.

Example Use Case

If entity B adds a second entry barrier at 192.168.10.45, the admin selects entity B, enters the IP under Entry 2 IP, and saves. All entry requests will now be directed to the new barrier.

Each entity’s settings are independent, so entity A and entity B can have different devices and capacities.

System Maintenance

The Maintenance section provides administrators with tools to manage data storage and keep the system optimized. Each entity database can be managed separately to ensure smooth operation.

Key Features

  • Select Entity: Like Communication Settings, you first choose the (entity) to apply maintenance actions.
  • Delete Old Records:
    • Admins can specify a cutoff date and choose which records to remove.
    • Options include Sales Records, Staff Shift Records, and Barrier Access Logs.
    • This helps free database space and improve performance.
  • Database Backup:
    • Admins can download a backup of the selected entity’s database in .sql format.
    • This is useful for disaster recovery and safe storage.
  • Database Restore:
    • Admins can upload a previous backup file (.sql) and restore the database.
    • This is helpful when recovering lost data or resetting to a stable state.

Example Use Case

If entity A’s database has grown too large, the admin can open System Maintenance, select entity A, choose Delete Sales Records before Jan 2024, and execute. This will clear all old sales data while keeping recent records intact.

Deleting records is permanent. Always perform a database backup before deletion.

Change Password

Every user can change their password for account security. Park Officers only see this option, while admins and managers also have access to Communication and Maintenance settings.
